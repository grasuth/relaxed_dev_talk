A Recipe for Relaxed Development

Meaning...

Development becomes productive, fun, progress is easy, there is a minimum of panic and chaos. A sense of doing it well and getting better over time.

Or wanting to show up to work despite a long commute.

Serves...

8-12 full time developers

1 QA Person

A couple of managers

Ingredients

  1. Scrum-style agile
  2. git and github
  3. modified github flow for branch/release
  4. 2 levels of code review
  5. separate QA
  6. 100% unit and functional test coverage
  7. pep8
  8. sphinx docs
  9. Rally (yawn)
  10. Jenkins
  11. Google docs burndown charts

Sounds like a lot of paperwork

But good tools help to keep things moving

Start development:

  1. create git branch for the whole story/bit of functionality
  2. create a dev branch for own work
  3. all development in individual dev branches

Development rules:

  1. PEP8 compliant
  2. Always docstrings
  3. High coverage of unit and functional tests

Code Review into Story Branch:

  1. set up in jenkins (pep8, tests, docs build)
  2. raise pull request into story branch
  3. somebody picks it up to review
  4. they check jenkins, check makes sense, comments, looks for anything that needs fixing
  5. developer revises and pushes, which shows changes in pull request.
  6. repeat until all good
  7. changes maybe required for architectural or other changes or because we are still working out how to do it, so no shame

Code Review into RC Review Branch:

When the story is finished, then it is reviewed again into release branch

  1. same review process - pull request into an rc_review branch
  2. reviewed by design authority/architect for overall architecture and whatever shows up. This is where some integration issues are resolved, so stuff can easily get rejected and reworked here.
  3. once in review branch, then available for QA

QA to Release:

  1. QA build and run own tests against complete stories and components
  2. Product owner, when happy, accepts story
  3. etc...

WINS:

  1. Very very low production bugs, maybe 1
  2. Confidently make massive changes to suit external system issues confidently.
  3. No need to be clever/experienced in everything. The process teaches you.
  4. Easy to matrix roles / step in / take over.
  5. etc...

Issues:

  1. Bottlenecks show up, so sprint by sprint the bottleneck moved around before firmly fixing in QA, resulting in QA Hothouses to clear stuff for release
  2. Burndown graph tools in Rally too stupid to reflect reality, so use google docs burndowns
  3. Too much paperwork, requires automating jenkins jobs, branch creations and naming
